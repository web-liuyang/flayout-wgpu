# 从一个图形到屏幕像素

这份文档用一个尽量具体的例子，手工推一遍：

**一个版图里的 shape，是怎么一步步变成屏幕上的像素的？**

如果你能跟着这份文档把坐标链路手推出来，当前 viewer 里最核心的几何和渲染思路你就真正掌握了。

## 我们选一个最简单的例子

假设有一个矩形：

- 世界坐标 bounds：`(0, 0) ~ (10, 20)`
- layer：`(1, 2)`

再假设当前相机和画布状态是：

- `zoom = 4.0`
- `pan = (100, 60)`
- `canvas_origin = (280, 0)`
- `viewport_size = (1160, 960)`

这个例子不一定和你当前真实窗口完全一致，
但足够用来说明整条坐标链。

## 第 1 步：shape 在导入层长什么样

在 `scene` 层，这个矩形大概会长成：

- `bounds = Bounds::new(0, 0, 10, 20)`
- `points = [(0,0), (10,0), (10,20), (0,20)]`
- `closed = true`
- `stroke_width_world = None`

注意这里还是**世界坐标**。

也就是说，此时它和屏幕完全没有关系。

## 第 2 步：为什么 renderer 先不直接算最终屏幕位置

你可能会很自然地想：

“那就直接算出最终屏幕坐标不就行了吗？”

但当前项目没有这么做，原因是：

- `pan` 会频繁变化
- 如果每次都把平移写死进顶点
- 平移一下，所有 tile buffer 都得重建

所以当前策略是：

- CPU 先只做 `world * zoom`
- shader 再加 `translation = pan + canvas_origin`

这样平移时更容易复用 tile buffer。

## 第 3 步：先做 `world * zoom`

现在看矩形的四个点：

- `(0, 0)`
- `(10, 0)`
- `(10, 20)`
- `(0, 20)`

乘上 `zoom = 4.0` 后得到：

- `(0, 0)`
- `(40, 0)`
- `(40, 80)`
- `(0, 80)`

这一步对应 `geometry.rs` 里的：

- `scaled_world_to_screen(...)`
- `build_scaled_line_vertices_for_indices(...)`

注意，这里得到的还不是最终屏幕位置。
它只是“缩放后的逻辑屏幕空间几何”。

## 第 4 步：为什么是“逻辑屏幕空间”

因为当前 viewer 同时要面对：

- 逻辑坐标（egui、交互）
- 物理像素（surface、scissor）

主渲染坐标链里，我们尽量在大部分几何计算阶段都使用逻辑坐标。
这样可以减少高 DPI 下的混乱。

所以你现在看到的这些 `(40, 80)` 之类的值，
更接近“逻辑点”而不是最终物理像素。

## 第 5 步：矩形为什么会变成 24 个顶点

矩形本来只有 4 条边，为什么最后测试里是 24 个顶点？

原因在 `emit_segment(...)`。

当前 viewer 不是依赖图形 API 的“原生线宽”，
而是自己把每一条边膨胀成两个三角形。

### 一条边会发生什么

假设一条边是：

- `start = (0, 0)`
- `end = (40, 0)`

如果线宽是 `1.5`，那么 renderer 会：

1. 算方向向量 `delta = end - start`
2. 算法线 `normal`
3. 用 `start/end ± normal` 得到 4 个角点
4. 组 2 个三角形

2 个三角形 = 6 个顶点

### 整个矩形

矩形有 4 条边：

- 4 × 6 = 24 个顶点

这就是测试里为什么会断言：

- `vertices.len() == 24`

## 第 6 步：什么时候才加 `pan + canvas_origin`

这一部分不是在 CPU 顶点生成里做的，
而是在 shader 里做。

当前例子里：

- `pan = (100, 60)`
- `canvas_origin = (280, 0)`

所以：

`translation = (380, 60)`

shader 会把前面缩放后的点：

- `(0, 0)`
- `(40, 0)`
- `(40, 80)`
- `(0, 80)`

统一加上这个 translation，变成：

- `(380, 60)`
- `(420, 60)`
- `(420, 140)`
- `(380, 140)`

这时它才真正落到当前窗口中的“画布位置”。

## 第 7 步：为什么还要加 `canvas_origin`

因为中央画布不是从整个窗口的 `(0, 0)` 开始画的。

左侧还有一个面板，占了大约 280 个逻辑点宽度。

所以即使 `pan = (0, 0)`：

- 版图也不应该从整个窗口左上角开始画
- 而应该从中央画布左上角开始画

这就是 `canvas_origin` 的意义：

**它把场景从“全窗口坐标系”移动到“中央画布坐标系”。**

## 第 8 步：最后怎么转成 NDC

GPU 顶点着色器最终要输出的是 NDC：

- x 在 `[-1, 1]`
- y 在 `[-1, 1]`

shader 里做的是：

- `ndc_x = (screen.x / viewport_width) * 2 - 1`
- `ndc_y = 1 - (screen.y / viewport_height) * 2`

继续用我们的例子：

假设某个点的最终屏幕坐标是：

- `(380, 60)`

而 viewport 是：

- `1160 x 960`

那么：

- `ndc_x = (380 / 1160) * 2 - 1 ≈ -0.3448`
- `ndc_y = 1 - (60 / 960) * 2 = 0.875`

这就是 GPU 最终真正拿来光栅化的坐标。

## 第 9 步：为什么 y 方向公式是反的

因为屏幕坐标通常是：

- 左上角是 `(0, 0)`
- 往下 y 变大

而 NDC 坐标通常是：

- 中心是 `(0, 0)`
- 往上 y 变大

所以 y 需要翻转：

`1 - (...)`

这是所有 2D 渲染坐标变换里一个很常见的点。

## 第 10 步：像素是怎么真正被填出来的

到 NDC 为止，还只是“顶点位置”。

接下来 GPU 会自动完成：

1. 组装三角形
2. 光栅化
3. 覆盖对应像素
4. 调 fragment shader 输出颜色

当前 fragment shader 非常简单：

- 直接返回顶点颜色

所以对于这个 viewer 来说，像素颜色几乎全由 CPU 决定：

- layer -> `layer_color(...)`

shader 本身没有做复杂的材质或光照计算。

## 第 11 步：为什么有时你看到的是轮廓而不是填充面

因为当前 renderer 对很多图元的策略是：

- 把边界当成线段轮廓画
- 不做真正的多边形填充

这就是为什么：

- `Boundary` 现在更像 outline
- `Path` 更像有宽度的线段

这是当前 viewer 有意保持简单的地方。

## 第 12 步：如果换成 path，会多什么步骤

如果 shape 是 path，而不是 rectangle：

- 它会带一个 `stroke_width_world`
- renderer 会先按当前 `zoom` 把世界线宽换成屏幕线宽
- 再对每一段 path 中心线调用 `emit_segment(...)`

也就是说，多出来的关键步骤只有一个：

**世界线宽 -> 屏幕线宽**

这也是为什么 path 没有在导入层直接烘焙成固定三角形。

## 第 13 步：为什么平移不会让 tile buffer 重建

回到当前例子，如果用户只做平移：

- `zoom` 不变
- world 点不变
- `world * zoom` 的结果不变

变的只是：

- `translation = pan + canvas_origin`

而这个 translation 现在在 shader uniform 里。

所以 tile 顶点本身不需要重建。

这正是当前 tile cache 能在平移时发挥作用的根本原因。

## 第 14 步：如果用户缩放，会发生什么

如果用户缩放：

- `zoom` 变了
- `world * zoom` 的结果也变了

所以 tile 顶点就不再能直接复用。

这就是为什么当前项目里：

- 平移倾向于 cache hit
- 缩放更容易触发 tile misses

## 用一句话总结整条链

当前一个 shape 走到屏幕像素的链路是：

`Scene points -> 乘 zoom -> 按边生成三角形 -> shader 加平移 -> shader 转 NDC -> GPU 光栅化 -> 颜色写入屏幕`

## 建议你自己手推的练习

你可以拿纸自己算一遍这个最小例子：

1. 一个矩形 `(0,0) ~ (10,20)`
2. `zoom = 4`
3. `pan = (100,60)`
4. `canvas_origin = (280,0)`
5. `viewport = (1160,960)`

你要自己算出：

- 缩放后的 4 个点
- 加平移后的 4 个点
- 左上角点的 NDC
- 为什么最后是 24 个顶点

如果这些你都能手算出来，当前 viewer 的几何主链你就已经真正吃透了。
